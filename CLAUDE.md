## Development

```bash
# Install dependencies
npm install && cd client && npm install && cd ..

# Start dev (API port 3000 + client port 5173)
npm run dev:all

# Individual services
npm run dev              # API server only
npm run client:dev       # Client only

# Testing
npm test                 # Server tests (watch mode)
npm run test:unit        # Server unit tests (single run)
npm run test:all         # All tests ‚Äî server + client
npm run test:coverage    # Coverage report
npx vitest run --reporter=basic  # 740 server tests in ~5s

# Build & lint
npm run build            # Production build
npm run lint             # ESLint

# Architecture contract
npm run arch:scan        # Regenerate schemas/arch.json
npm run arch:check       # Drift check vs committed schemas/arch.json (CI runs this)

# Database
npm run db:migrate       # Apply Supabase migrations (supabase db push)

# Optional Python ML engine
npm run ml:start         # uvicorn on port 8000
```

This section is the operator's runbook for keeping Frontier Alpha green. It assumes you have `vercel`, `railway`, and `gh` CLIs authenticated, and that you're working from a clone of this repo.

### Daily check (5-minute ritual)

1. **Hit `/api/v1/health/integrations`** ‚Äî `curl -s https://frontier-alpha.metaventionsai.com/api/v1/health/integrations | jq '.integrations | to_entries[] | select(.value.status != "live") | .key'`. Anything other than `polygonWebSocket` (Vercel by-design degraded) is a real signal.
2. **Synthetic monitor status** ‚Äî `curl -s "https://frontier-alpha.metaventionsai.com/api/v1/cron/synthetic-monitor?key=$CRON_SECRET" | jq '.data | {passed, failed, results: [.results[] | select(.error)]}'`. US-007 wires this against the same shape assertions as `tests/integration/protected-routes.test.ts` (see "Golden state fixture" below). Status budget: `failed: 0`. Failures auto-increment `errorCounter` so `/api/v1/health/errors` and the weekly digest also surface them. Vercel cron is configured to hit it every 15 minutes ‚Äî for manual debugging, the same query with `?key=$CRON_SECRET` returns the latest pass/fail per route. Also worth a glance: `curl -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" https://frontier-alpha.metaventionsai.com/api/v1/health/errors | jq '.data.totalErrors'` ‚Äî should be 0 most days.
3. **Eyeball production** ‚Äî `https://frontier-alpha.metaventionsai.com/dashboard` in a fresh-incognito session signed in as `dicoangelo+dev@metaventionsai.com`. No console errors, no "Reconnecting ¬∑ 1 ¬∑ 1s" banner that doesn't terminate, no $125K mock numbers leaking through.

### Visual regression (TOKEN-007)

Playwright snapshot suite that diffs every primary page (Landing, Dashboard, Portfolio, Trading, Options, CVRF, Earnings, Factors, Alerts) against the committed baseline in both light and dark themes at 1440x900. Runs **nightly only** (06:00 UTC via `.github/workflows/visual-regression.yml`), not per PR. Story rationale: a 20-minute Playwright pass on every PR is not worth the cycle cost when the no-raw-hex ESLint rule (TOKEN-006) already gates the migration and the visual diff is a backstop, not a primary check.

```bash
# Run against committed baselines (requires SUPABASE_* env for golden-state user)
npm run test:visual

# Regenerate baselines after intentional design changes
npm run test:visual:update
```

- **Config:** `tests/visual/playwright.config.cjs` (CommonJS on purpose; the repo root is `"type": "module"` and the playwright CJS-to-ESM bridge can't surface `defineConfig` through Node 22's static analysis when the config is ESM. The local `tests/visual/package.json` pins this directory to CJS so the spec files load via Playwright's require() path.)
- **Spec:** `tests/visual/tokens.spec.ts` ‚Äî 18 tests = 9 pages x 2 themes.
- **Baselines:** `tests/visual/baseline/*.png` ‚Äî git-tracked PNG, lossless. Diff tolerance is `maxDiffPixelRatio: 0.01` (1% per the story).
- **Auth:** Reuses the same seeded `dicoangelo+test@metaventionsai.com` golden-state user as the integration suite (5 holdings, 3 alerts, 2 realized lots, enterprise comp). The visual auth helper at `tests/visual/auth-helper.ts` is a deliberate copy of `tests/integration/auth-helper.ts::mintTestSession` because the two test suites live in different module scopes (CJS vs ESM); both must stay in sync if the seeded user changes.
- **Theme toggle:** the spec writes the Zustand persist key `frontier-theme` to localStorage before navigation so the theme is applied at first paint, no FOUC.
- **Wait anchors:** every page exposes a `data-testid="visual-{page}-ready"` element on its loaded-state branch. If you add a new page or restructure an existing one, add the test ID before regenerating baselines.
- **CI failure artifacts:** the workflow uploads `playwright-report/` and `test-results/` (both root-level and `tests/visual/`) on any diff > 1%. Pull them, eyeball the diff PNG, decide: real regression vs. baseline staleness from an intentional design change.

To regenerate baselines after a design change:

```bash
# 1. Make the design change.
# 2. Run with --update-snapshots:
npm run test:visual:update
# 3. Eyeball every changed PNG in tests/visual/baseline/.
# 4. Commit the new baselines as a separate commit so the diff is reviewable.
```

CI requires the same secrets as the integration tests (`SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `SUPABASE_ANON_KEY` or `VITE_SUPABASE_ANON_KEY`, plus `VITE_SUPABASE_URL`). The nightly workflow reads them from repo secrets.

**To seed manually** (e.g., before a sales screenshot session or to repro a customer bug):

```bash
# From project root, with .env loaded (SUPABASE_URL + SUPABASE_SERVICE_KEY + SUPABASE_ANON_KEY)
INTEGRATION=true npx vitest run tests/integration --reporter=basic
# OR programmatically: import { mintTestSession, seedGoldenState } from 'tests/integration/auth-helper.ts'
```

The fixture is **idempotent** (delete-then-insert keyed on user_id), so running twice yields the same state. The integration suite seeds in `beforeAll` and clears in `afterAll`. The SQL fallback at `tests/fixtures/golden-state.sql` documents the schema-level contract for operators piping through `psql` directly. **Both must stay in sync** ‚Äî if you change one, change the other.

**Why this exists:** before US-007, every smoke test, sales screenshot, and support repro invented its own fixture. Now there's one canonical state, machine-checkable schemas (`schemas/api-shape.json`) for what the protected routes return against it, and the same assertions run in CI (`npm run test:integration`) AND production (`/api/v1/cron/synthetic-monitor`).

### Weekly check

1. **Env audit** ‚Äî `npm run env:audit` (US-009). Pulls Vercel production envs, validates against `schemas/env-schema.json` (length, regex, trailing whitespace, embedded `\n`, quoted-empty), reports `‚úì clean` / `‚öÝ suspicious` / `‚úó corrupted`, and exits 1 on any `‚úó`. Always cleans up `.env.audit`. Best-effort Railway audit appended via `railway variables --json` when the CLI supports it. CI also runs this on a Monday 12:00 UTC cron via `.github/workflows/env-audit.yml` (needs the `VERCEL_TOKEN` repo secret).
2. **Schema-only sweep** ‚Äî `npm run env:audit:schema` (US-009). No Vercel access needed; greps every `process.env.X` and `import.meta.env.X` reference and fails if any var is missing from `schemas/env-schema.json`. Same check runs on every PR via the `env-audit.yml` workflow's `schema-only` job ‚Äî adding a new `process.env.X` without a schema entry fails the PR.
3. **Sentry / error-rate digest** ‚Äî US-008 (v1.3.0) wires the weekly cron at `GET /api/v1/digest/health-summary`, scheduled Sundays 13:00 UTC via `vercel.json`. The email goes to `dicoangelo@metaventionsai.com` and aggregates: error counts per route (in-process `errorCounter`, 1-hour rolling window), integration health summary from `/health/integrations`, cache hit ratio (placeholder pending US-006), and the latest deploy id. Sentry status: server-side `SENTRY_DSN` is currently UNSET (only `client/src/lib/sentry.ts` has scaffolding via `VITE_SENTRY_DSN`); the digest body flags this so the operator knows the in-process counter is the sole source. **TODO (v1.4):** when `SENTRY_DSN` lands, query the Sentry Issues API in the digest cron ‚Äî `GET https://sentry.io/api/0/projects/{org}/{project}/issues/?statsPeriod=7d&query=is:unresolved` ‚Äî and merge those counts into the email so the digest reflects the full week, not just the current 1-hour counter window. The query call site is `src/routes/health-summary.ts::healthSummaryRoutes` (see the `sentryConfigured` flag).
4. **Deploy log scan** ‚Äî `vercel logs frontier-alpha --since 7d | grep -iE "error|fatal|cold start|timeout"`. Cold-start anomalies often precede env-corruption surprises.
5. **Run `npm run arch:check`** locally to confirm `schemas/arch.json` matches HEAD. CI does this on every PR but a periodic manual check catches drift introduced by rebased branches.

### Incident playbook ‚Äî `/health/integrations` reports degraded

When an integration flips to `degraded` or `offline`, walk this ladder in order. Don't skip steps; don't rotate keys before confirming the upstream is actually up.

1. **Provider status check** ‚Äî Polygon (`status.polygon.io`), Stripe (`status.stripe.com`), Supabase (`status.supabase.com`), Resend (`status.resend.com`), Alpha Vantage (no formal status page; check Hacker News). If the provider itself is down, mark and wait.
2. **Compare env** ‚Äî `vercel env pull .env.tmp --environment=production`, then `grep -nE "^${PROVIDER_VAR}=" .env.tmp` to see the literal value Vercel will hand the runtime. Look for trailing `\n`, leading/trailing whitespace, embedded quotes, or length truncation. Compare to your password manager / source of truth. (Polygon keys are 32 chars; Stripe live keys start with `sk_live_`; Supabase service keys are JWTs.)
3. **Test upstream directly** ‚Äî bypass Frontier Alpha entirely. For Polygon: `curl -i "https://api.polygon.io/v2/aggs/ticker/AAPL/prev?apiKey=$KEY"`. For Stripe: `curl -i -u $KEY: https://api.stripe.com/v1/balance`. For Supabase: `curl -i -H "apikey: $KEY" "https://${PROJECT}.supabase.co/rest/v1/"`. If the curl works but the app fails, the env value in Vercel is corrupt ‚Äî go to step 4. If the curl fails too, the key is wrong or revoked ‚Äî rotate.
4. **Rotate** ‚Äî see "Rotation runbook" below. Always `printf`, never `echo`.
5. **Redeploy** ‚Äî `vercel --prod` (or `gh workflow run deploy.yml`). Wait 60s for Vercel cache to invalidate.
6. **Verify** ‚Äî re-curl `/api/v1/health/integrations` and confirm the entry flips back to `live` (US-004 will add a 60s probe-result cache, so allow up to 60s).

### Rotation runbook

The `printf-not-echo` rule exists because `echo` adds a newline by default and `vercel env add` consumes the value verbatim. A 32-char Polygon key arrives as 33 chars, the upstream reads 32, the 33rd is dropped, and the resulting key 401s. Captured as `feedback_polygon_key_truncation.md`. **Always**:

```bash
# WRONG ‚Äî adds trailing newline, corrupts the env
echo "abc123..." | vercel env add POLYGON_API_KEY production --force

# RIGHT ‚Äî printf %s prints exactly what you give it, no newline
printf "%s" "abc123..." | vercel env add POLYGON_API_KEY production --force

# Also right ‚Äî pasting via stdin
vercel env add POLYGON_API_KEY production --force
# (then paste the key followed by Enter; only safe if you trust your terminal not to add a CR)
```

For each provider:

| Provider | Where to rotate | Env var |
|---|---|---|
| Polygon | `https://polygon.io/dashboard/api-keys` | `POLYGON_API_KEY` (32 chars) |
| Stripe | `https://dashboard.stripe.com/apikeys` | `STRIPE_SECRET_KEY` (`sk_live_...` or `sk_test_...`), `STRIPE_WEBHOOK_SECRET` (rotate via webhook endpoint resend) |
| Supabase | Project Settings ‚Üí API | `SUPABASE_SERVICE_KEY`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` (paired) |
| Resend | `https://resend.com/api-keys` | `RESEND_API_KEY` |
| DeepSeek / OpenAI | provider dashboards | `DEEPSEEK_API_KEY`, `OPENAI_API_KEY` |
| VAPID | regenerate via `node scripts/wire-vapid.sh` | `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY` (paired) |
| Cron / encryption | regenerate locally with `openssl rand -hex 32` | `CRON_SECRET` (64 hex), `BROKER_CRED_ENC_KEY` (64 hex) |

After every rotation: mirror the value to **Railway** (the always-on tier) via `railway variables set NAME="value"` (Railway accepts the raw value without the printf dance because it doesn't pipe through stdin). Redeploy both tiers. Re-curl `/api/v1/health/integrations` to confirm.

The v1.3.0 US-009 ships `scripts/audit-env.mjs` + `schemas/env-schema.json` so corruption is caught at PR time and rotations get partial automation.
